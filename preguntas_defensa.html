Respuesta de Seguridad

Ante la pregunta: ¿Su sistema predice cuándo va a ocurrir un sismo?. Responde con firmeza: No. Ningún modelo en el mundo predice el tiempo exacto de un sismo. Nuestro proyecto estima el potencial de magnitud esperado en el espacio (Random Forest) y la **probabilidad geográfica de acumulación** (KDE) para mitigar riesgos.

General y ETL
1. ¿Cuál es la pregunta de negocio principal y por qué es relevante?
▾

Pregunta central: ¿Cómo identificar y predecir las zonas de mayor riesgo sísmico en Ecuador para optimizar la asignación de recursos de prevención y respuesta?

Relevancia: Ecuador se sitúa en el Cinturón de Fuego del Pacífico (zona de subducción). Identificar hotspots permite salvar vidas mediante simulacros focalizados, mejorar normas de construcción y asignar brigadas con eficiencia presupuestaria.

General y ETL
2. ¿De dónde provienen los datos y qué periodo de tiempo abarcan?
▾

Fuente: Catálogo oficial del Instituto Geofísico de la Escuela Politécnica Nacional (IG-EPN).

Periodo: Del año 2012 al 2025 (13 años de registros históricos), consolidando un dataset final de 3,369 sismos válidos.

General y ETL
3. ¿Qué transformaciones clave se aplicaron en el pipeline de ETL?
▾
  • Conversión de tipos: Fechas a `datetime` y coordenadas/profundidades/magnitud a flotantes (`float64`).
  • Imputación de magnitud: Relleno de datos nulos unificando `magnitude_value_M` con `P`.
  • Variables derivadas: Región (Norte, Centro, Sur) según la latitud; Categorías de Magnitud (Ligero/Moderado/Fuerte) y Profundidad (Superficial/Intermedio/Profundo).
  • Limpieza: Eliminación estricta de valores nulos en columnas críticas y exportación a Parquet.
General y ETL
4. ¿Por qué se utilizó el formato Parquet en lugar de CSV para los datos procesados?
▾

Por rendimiento y optimización de recursos:

  • Almacenamiento: Almacena de forma columnar comprimida, reduciendo espacio en disco.
  • Preservación del esquema: Mantiene los tipos de datos exactos sin necesidad de redefinirlos al leerlos.
  • Carga ultra rápida: Acelera drásticamente la recarga de datos en Streamlit al usar `@st.cache_data`.
General y ETL
5. ¿Qué sesgos presenta el dataset y cómo se mitigaron?
▾
  • Sesgo instrumental: Sismos recientes leves parecen más frecuentes por la mejora histórica de sismógrafos. Se mitiga filtrando sismos menores (magnitud ≥ 3.5) en análisis de riesgo sísmico.
  • Sesgo físico: Ausencia de variables de suelo o fallas. Se mitiga definiendo el alcance como un modelo netamente probabilístico e histórico, no de simulación física en tiempo real.
Dashboard y API
6. ¿Cómo se estructuró la arquitectura de software del proyecto?
▾

Es una arquitectura modular e independiente dividida en 3 capas:

  • Datos (ETL): Modulo (`process_data.py`) que procesa y escribe el archivo Parquet.
  • Backend (FastAPI): API REST (`api.py`) que expone los datos y KPIs para clientes externos.
  • Frontend (Streamlit): Dashboard interactivo (`app.py`) con visualizaciones en Plotly, mapas y ejecución en caliente del modelo de predicción.
Dashboard y API
7. ¿Por qué son necesarios `@st.cache_data` y `@st.cache_resource` en Streamlit?
▾

Porque Streamlit vuelve a ejecutar todo el código de arriba a abajo ante cada interacción del usuario. El almacenamiento en caché evita retardos:

  • `@st.cache_data`: Almacena en memoria resultados de lectura de datos y cálculos puros (ej. DataFrame de sismos y capas de mapas).
  • `@st.cache_resource`: Mantiene persistente en memoria el modelo cargado de Random Forest (`.joblib`), evitando inicializarlo repetidamente.
Dashboard y API
8. ¿Cómo interactúa Streamlit con el mapa dinámico de Folium?
▾

Utiliza la librería `streamlit-folium`:

  • Se dibuja un mapa base con marcadores circulares de sismos.
  • El tamaño es proporcional a su magnitud (`radio = mag * 1.5`) y el color indica la categoría de alerta (verde, naranja, rojo).
  • El componente retorna a Streamlit un diccionario interactivo; cuando detecta un clic en el mapa, captura las coordenadas lat/lon del evento.
Dashboard y API
9. ¿Cómo funciona la reactividad del simulador de sismos?
▾

Sincroniza coordenadas en tiempo real:

  • El clic del usuario en el mapa actualiza las variables en `st.session_state` y dispara `st.rerun()`.
  • Los sliders de entrada se alinean automáticamente con las coordenadas seleccionadas.
  • El backend pasa estos datos instantáneamente al modelo Random Forest y actualiza en pantalla la magnitud estimada y el radio de afectación.
Dashboard y API
10. ¿Cómo escalaría la API de FastAPI para soportar millones de registros?
▾
  • Base de Datos Espacial: Conectando a una base PostgreSQL/PostGIS para realizar filtros geográficos indexados.
  • Paginación: Restringiendo el número de resultados mediante parámetros de `limit` y `offset`.
  • Asincronía: Declarando los endpoints como `async def` para permitir concurrencia de peticiones no bloqueantes.
Dashboard y API
11. ¿Qué es el Middleware de CORS en la API y para qué sirve?
▾

CORS (Cross-Origin Resource Sharing): Mecanismo de seguridad del navegador.

Se habilitó en FastAPI (`allow_origins=["*"]`) para permitir que la web de Streamlit, u otra aplicación cliente externa en un dominio distinto, acceda a los datos de la API en formato JSON sin ser bloqueados.

Random Forest
12. ¿Qué significa y cómo funciona el modelo Random Forest Regressor?
▾
Definición de Random Forest Regressor

Es un modelo de aprendizaje supervisado de ensamble que combina múltiples árboles de decisión independientes. Su predicción final es el promedio de todas las respuestas del "bosque", reduciendo el error y previniendo el sobreajuste.

Funcionamiento: Durante el entrenamiento, el algoritmo crea una colección de árboles de decisión (bosque). Cada árbol se genera a partir de una muestra aleatoria del dataset (Bagging) y selecciona particiones de variables de manera aleatoria. Al realizar una predicción, promedia los resultados de todos los árboles para estimar el valor final (magnitud del sismo).

Random Forest
13. ¿Cuáles son las variables predictoras (entradas) y el target (salida) del modelo?
▾
  • Variables Predictoras (Features / Entrada): Latitud (`lat`), Longitud (`lon`) y Profundidad (`depth`).
  • Variable Objetivo (Target / Salida): Magnitud del sismo (`magnitude`).
Random Forest
14. ¿Qué métricas se utilizaron para evaluar este regresor?
▾
  • MAE (Error Absoluto Medio): Promedio de la diferencia absoluta entre predicción y realidad. Indica cuánto falla el modelo (ej. ±0.4 Mw de magnitud).
  • MSE (Error Cuadrático Medio): Promedio de errores al cuadrado. Penaliza drásticamente los fallos grandes.
  • R² (Coeficiente de determinación): Indica el porcentaje de variabilidad de la magnitud explicada por la localización física del sismo.
Random Forest
15. Físicamente, ¿por qué es difícil estimar la magnitud de un sismo usando solo su ubicación?
▾

Físicamente la magnitud depende de factores dinámicos no geográficos:

  • La longitud física de la ruptura en la falla tectónica.
  • La energía de esfuerzo elástico acumulada y fricción del suelo local.
Aclaración metodológica: El modelo no simula física de ondas; calcula una estimación histórica espacial. Aprende qué magnitudes son esperables en ciertas coordenadas y profundidades según el registro de eventos históricos locales.
Random Forest
16. ¿Qué hiperparámetros son clave para regular el sobreajuste (overfitting)?
▾
  • `max_depth`: Limita la profundidad de los árboles para evitar que memoricen el ruido de los sismos de entrenamiento.
  • `min_samples_split`: Mínimo de sismos requeridos en un nodo para dividirse en ramas.
  • `n_estimators`: Cantidad de árboles en el bosque (más árboles estabilizan la predicción, pero requieren mayor cómputo).
Random Forest
17. ¿Cómo se integró el modelo entrenado en la interfaz de Streamlit?
▾

Se exportó el modelo entrenado a formato binario usando `joblib`. En la app:

  • Se carga una sola vez en memoria mediante el decorador de caché `@st.cache_resource`.
  • Al recibir las coordenadas (`lat`, `lon`, `depth`) del usuario, se ejecuta `predict()` en tiempo real.
  • El dashboard actualiza instantáneamente los indicadores visuales y el radio de afectación.
Modelo KDE
18. ¿Qué significa y cuál es el objetivo de la Estimación de Densidad Kernel (KDE)?
▾
Definición de KDE (Kernel Density Estimation)

Es un método estadístico no paramétrico que calcula la densidad de probabilidad continua de un conjunto de puntos. En cartografía, suaviza los sismos individuales convirtiéndolos en un mapa de color continuo (calor) para revelar hotspots de riesgo.

Objetivo: Identificar y visualizar las zonas de mayor concentración de sismicidad histórica en Ecuador. Facilita la toma de decisiones al delimitar las zonas críticas sin verse sesgados por límites políticos cantonales o provinciales.

Modelo KDE
19. ¿Qué es el Bandwidth (ancho de banda) en KDE y cómo afecta al mapa de riesgo?
▾

Es el parámetro que define el radio de suavizado de cada punto:

  • Bandwidth pequeño: Genera picos de calor muy pequeños y aislados en cada sismo exacto (mapa ruidoso, sin tendencias claras).
  • Bandwidth grande: Diluye los hotspots en una mancha gigante y borrosa, perdiendo el detalle geográfico de dónde se concentran las fallas.
  • Selección: Usamos valores entre 0.3 y 0.5 grados para un equilibrio óptimo de visualización regional.
Modelo KDE
20. ¿Por qué es importante utilizar la distancia de Haversine al hacer KDE geográfico?
▾

Porque la Tierra es una esfera. Si calculamos distancias euclidianas planas sobre grados de latitud/longitud, se produce distorsión geométrica.

La métrica de Haversine calcula la distancia real en kilómetros entre coordenadas esféricas del globo terrestre, garantizando que el radio del kernel de riesgo sea físicamente exacto.

Modelo KDE
21. ¿Cómo se transformaron las densidades abstractas del KDE en niveles de riesgo?
▾

Se calculó la densidad del kernel sobre los sismos filtrados e históricos, normalizando los valores en una escala de 0 a 1. Se clasificó en 3 umbrales lógicos de riesgo:

  • Riesgo Bajo (Amarillo / < 0.33): Zonas con baja recurrencia de sismos históricos.
  • Riesgo Moderado (Naranja / 0.33 a 0.66): Zonas con sismicidad periódica constante.
  • Riesgo Alto (Rojo / ≥ 0.66): Hotspots con acumulación crítica de sismos históricos (ej. fallas costeras e interandinas).
Modelo KDE
22. ¿Cómo ayuda el mapa KDE a las entidades gubernamentales de control?
▾
  • Infraestructura y Prevención: Permite planificar el despliegue estratégico de centros de rescate, hospitales de campaña y reservas de ayuda en las áreas de mayor riesgo.
  • Planificación Territorial: Identifica áreas donde se deben exigir normas de construcción sismorresistentes más rigurosas de forma prioritaria.
Tips de Defensa
23. ¿Cómo defender el modelo regresor ante jurados que cuestionen su fiabilidad física?
▾

Respuesta sugerida: "Es cierto que la localización física no causa directamente la magnitud de un sismo. Por ello, el modelo no opera como predictor físico en tiempo real, sino como un **estimador empírico e histórico**. Evalúa la sismicidad registrada localmente en el pasado para predecir la **magnitud esperable** en caso de activarse un sismo en esa coordenada y profundidad, sirviendo para planificar escenarios hipotéticos de riesgo."

Tips de Defensa
24. ¿Cuál fue el principal reto en el desarrollo del software?
▾

Optimizar el rendimiento en la renderización de mapas de Streamlit. Al reejecutarse el script con cada slider, cargar miles de marcadores saturaba la UI. Se solucionó implementando **caché inteligente** (`@st.cache_data` y `@st.cache_resource`) y muestreando datos dinámicamente para lograr respuestas inmediatas en milisegundos.

Tips de Defensa
25. Si tuvieran más tiempo, ¿qué mejoras implementarían en los modelos?
▾
  • Enriquecimiento geológico: Integrar variables de fallas geológicas activas y tipos de suelo como entradas del Random Forest.
  • Modelamiento temporal: Utilizar modelos como procesos de Hawkes o redes LSTM para evaluar la probabilidad temporal de réplicas.
  • Ingesta automatizada: Programar tareas automáticas de web scraping para alimentar el dataset desde el IG-EPN en tiempo real.
